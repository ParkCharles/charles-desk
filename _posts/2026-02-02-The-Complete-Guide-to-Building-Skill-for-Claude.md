## 소개

**스킬(Skill)**은 Claude에게 특정 작업이나 워크플로우를 처리하는 방법을 가르치는 일련의 지침으로, 간단한 폴더 형태로 패키징됩니다. 스킬은 Claude를 특정 요구사항에 맞게 커스터마이징하는 가장 강력한 방법 중 하나입니다. 매번 대화에서 선호사항, 프로세스, 도메인 전문성을 재설명하는 대신, 스킬을 통해 한 번 가르치면 모든 대화에서 혜택을 받을 수 있습니다.
스킬은 반복 가능한 워크플로우가 있을 때 강력합니다: 스펙에서 프론트엔드 디자인 생성, 일관된 방법론으로 연구 수행, 팀의 스타일 가이드를 따르는 문서 생성, 또는 다단계 프로세스 오케스트레이션 등. Claude의 코드 실행 및 문서 생성과 같은 내장 기능과 잘 작동합니다. MCP 통합을 빌드하는 경우, 스킬은 원시 도구 접근을 안정적이고 최적화된 워크플로우로 전환하는 데 도움이 되는 강력한 추가 레이어입니다.
이 가이드는 계획 및 구조부터 테스트 및 배포까지 효과적인 스킬을 빌드하는 데 필요한 모든 것을 다룹니다. 자신을 위해, 팀을 위해, 또는 커뮤니티를 위해 스킬을 빌드하든, 실용적인 패턴과 실제 예제를 찾을 수 있을 것입니다.

### 배울 내용

- 스킬 구조에 대한 기술적 요구사항과 모범 사례
- 독립형 스킬과 MCP 강화 워크플로우를 위한 패턴
- 다양한 사용 사례에서 효과적으로 작동하는 패턴
- 스킬을 테스트, 반복, 배포하는 방법

### 이 가이드의 대상

- Claude가 특정 워크플로우를 일관되게 따르도록 원하는 개발자
- Claude가 특정 워크플로우를 따르도록 원하는 파워 유저
- 조직 전체에서 Claude의 작동 방식을 표준화하려는 팀

### 이 가이드의 두 가지 경로

**독립형 스킬**: 기초, 계획 및 설계, 카테고리 1-2에 집중하세요.

두 경로 모두 동일한 기술적 요구사항을 공유하지만, 자신에게 관련된 것을 선택하면 됩니다.

### 이 가이드에서 얻을 것

마지막까지 완료하면 단 한 번의 작업으로 기능적인 스킬을 빌드할 수 있을 것입니다. 템플릿을 사용하여 첫 번째 작동하는 스킬을 빌드하고 테스트해보세요.

## 1장: 기초

### 스킬이란 무엇인가?

스킬은 다음을 포함하는 폴더입니다:

- **SKILL.md** (필수): YAML 프론트매터가 포함된 Markdown 지침
- **scripts/** (선택): 실행 가능한 코드 (Python, Bash 등)
- **references/** (선택): 필요에 따라 로드되는 문서
- **assets/** (선택): 출력에 사용되는 템플릿, 폰트, 아이콘

### 핵심 설계 원칙

#### 점진적 공개 (Progressive Disclosure)

스킬은 세 가지 레벨 시스템을 사용합니다:

1. **첫 번째 레벨** (YAML 프론트매터)
   - 항상 Claude의 시스템 프롬프트에 로드
   - 각 스킬이 언제 사용되어야 하는지 알 수 있는 충분한 정보만 제공

2. **두 번째 레벨** (SKILL.md 본문)
   - Claude가 현재 작업과 관련이 있다고 생각할 때 로드
   - 전체 지침과 안내가 포함

3. **세 번째 레벨** (링크된 파일)
   - 스킬 디렉토리 내에 번들로 포함된 추가 파일
   - Claude가 필요에 따라 선택하여 탐색 가능

이 점진적 공개는 전문화된 지식을 유지하면서 토큰 사용을 최소화합니다.
#### 구성 가능성 (Composability)

Claude는 여러 스킬을 동시에 로드할 수 있습니다. 스킬은 다른 스킬과 함께 잘 작동해야 하며, 유일한 기능이라고 가정해서는 안 됩니다.

#### 이식성 (Portability)

스킬은 Claude.ai, Claude Code, API에서 동일하게 작동합니다. 스킬을 한 번 생성하면 스킬의 종속성을 지원하는 환경이라면 수정 없이 모든 표면에서 작동합니다.

---

### MCP 빌더를 위한: 스킬 + 커넥터

> **팁**: MCP 없이 독립형 스킬을 빌드하나요? [계획 및 설계](#2장-계획-및-설계) 섹션으로 건너뛰세요 - 나중에 여기로 돌아올 수 있습니다.

이미 작동하는 MCP 서버가 있다면 어려운 부분은 이미 끝난 것입니다. 스킬은 그 위에 지식 레이어입니다. 이미 알고 있는 워크플로우와 모범 사례를 캡처하여 Claude가 이를 적용할 수 있도록 합니다.
#### 주방 비유

- **MCP**: 전문 주방을 제공합니다 (도구, 재료, 장비에 대한 접근)
- **스킬**: 레시피를 제공합니다 (무언가를 만드는 방법에 대한 단계별 지침)

함께 사용하면 사용자가 각 단계를 파악할 필요 없이 복잡한 작업을 완수할 수 있습니다.

#### 함께 작동하는 방식

| MCP (연결성) | 스킬 (지식) |
|---|---|
| Claude를 서비스에 연결 (Notion, Asana, Linear 등) | Claude에게 서비스를 효과적으로 사용하는 방법을 가르침 |
| 실시간 데이터 접근 및 도구 호출 제공 | 워크플로우와 모범 사례 캡처 |
| Claude가 **무엇을 할 수 있는지** | Claude가 **어떻게 해야 하는지** |

#### MCP 사용자에게 중요한 이유

**스킬 없을 때:**
- 사용자는 MCP에 연결했지만 다음에 무엇을 해야 할지 모름
- "귀하의 통합으로 X를 어떻게 하나요?"라는 지원 티켓
- 각 대화가 처음부터 시작
- 사용자가 매번 다르게 프롬프트하기 때문에 일관되지 않은 결과

**스킬 있을 때:**
- 필요할 때 자동으로 활성화되는 사전 빌드된 워크플로우
- 일관되고 안정적인 도구 사용
- 모든 상호작용에 포함된 모범 사례
- 통합에 대한 낮은 학습 곡선

## 2장: 계획 및 설계

### 사용 사례로 시작하기

코드를 작성하기 전에 스킬이 처리해야 할 2-3가지 구체적인 사용 사례를 식별하세요.

#### 좋은 사용 사례 정의 예

**사용 사례**: 프로젝트 스프린트 계획

**트리거**: 사용자가 "이 스프린트를 계획해줘" 또는 "스프린트 작업 생성"이라고 말함

**단계:**
1. Linear에서 현재 프로젝트 상태 가져오기 (MCP 통해)
2. 팀 속도 및 용량 분석
3. 작업 우선순위 제안
4. 적절한 라벨과 추정치로 Linear에 작업 생성

**결과**: 작업이 생성된 완전히 계획된 스프린트
#### 스스로에게 물어보세요

1. 사용자가 무엇을 성취하고 싶어하나요?
2. 이를 위해 어떤 다단계 워크플로우가 필요한가요?
3. 어떤 도구가 필요한가요 (내장 또는 MCP)?
4. 어떤 도메인 지식이나 모범 사례를 포함해야 하나요?

### 일반적인 스킬 사용 사례 카테고리

Anthropic에서 관찰한 세 가지 일반적인 사용 사례:
#### 카테고리 1: 문서 및 자산 생성

**사용 목적**: 문서, 프레젠테이션, 앱, 디자인, 코드 등 일관되고 고품질의 출력 생성

**실제 예**: frontend-design 스킬

> "높은 디자인 품질의 차별화된 프로덕션급 프론트엔드 인터페이스를 생성합니다. 웹 컴포넌트, 페이지, 아티팩트, 포스터 또는 프로토타입을 빌드할 때 사용합니다."

**핵심 기법:**
- 임베디드 스타일 가이드 및 브랜드 표준
- 일관된 출력을 위한 템플릿 구조
- 최종 확정 전 품질 체크리스트
- 외부 도구 불필요 - Claude의 내장 기능 사용
#### 카테고리 2: 워크플로우 자동화

**사용 목적**: 일관된 방법론에서 이점을 얻는 다단계 프로세스, 여러 MCP 간 조정 포함

**실제 예**: skill-creator 스킬

> "새로운 스킬 생성을 위한 대화형 가이드. 사용 사례 정의, 프론트매터 생성, 지침 작성, 테스트를 안내합니다."

**핵심 기법:**
- 검증 게이트가 있는 단계별 워크플로우
- 일반적인 구조를 위한 템플릿
- 내장 리뷰 및 개선 제안
- 반복적 정제 루프
#### 카테고리 3: MCP 강화

**사용 목적**: MCP 서버가 제공하는 도구 접근을 강화하는 워크플로우 안내

**실제 예**: sentry-code-review 스킬 (Sentry 제공)

> "Sentry의 MCP 통합을 통해 오류 모니터링 데이터를 사용하여 GitHub Pull Request에서 감지된 버그를 자동으로 분석하고 수정합니다."

**핵심 기법:**
- 순차적으로 여러 MCP 호출 조정
- 도메인 전문성 임베드
- 사용자가 그렇지 않으면 지정해야 할 컨텍스트 제공
- 일반적인 MCP 문제에 대한 오류 처리

### 성공 기준 정의

#### 스킬이 작동하는지 어떻게 알 수 있나요?

이는 열망적인 목표입니다 - 정확한 임계값보다는 대략적인 벤치마크입니다. 엄격함을 목표로 하되, 일부 주관적 평가가 포함될 수 있음을 받아들이세요.

#### 정량적 지표

1. **스킬이 관련 쿼리의 90%에서 트리거됨**
   - 측정 방법: 스킬을 트리거해야 하는 10-20개의 테스트 쿼리를 실행
   - 자동으로 로드되는 횟수와 명시적 호출이 필요한 횟수 추적

2. **X 도구 호출로 워크플로우 완료**
   - 측정 방법: 스킬 활성화 여부와 관계없이 동일한 작업 비교
   - 도구 호출 및 총 토큰 소비량 계산

3. **워크플로우당 0개의 실패한 API 호출**
   - 측정 방법: 테스트 실행 중 MCP 서버 로그 모니터링
   - 재시도율과 오류율 추적

#### 정성적 지표

1. **사용자가 다음 단계에 대해 Claude에게 프롬프트할 필요 없음**
   - 평가 방법: 테스트 중 재지시 또는 명확화 필요 빈도 기록

2. **사용자 수정 없이 워크플로우 완료**
   - 평가 방법: 동일한 요청을 3-5번 실행하고 출력 비교

3. **세션 간 일관된 결과**
   - 평가 방법: 최소한의 안내로 새로운 사용자가 작업 완수 가능 여부

### 기술적 요구사항

#### 파일 구조

```
your-skill-name/
├── SKILL.md              # 필수 - 메인 스킬 파일
├── scripts/              # 선택 - 실행 가능한 코드
│   ├── process_data.py   # 예시
│   └── validate.sh       # 예시
├── references/           # 선택 - 문서
│   ├── api-guide.md      # 예시
│   └── examples/         # 예시
└── assets/               # 선택 - 템플릿 등
    └── report-template.md # 예시
```
#### 핵심 규칙

**SKILL.md 명명:**
- 정확히 `SKILL.md`여야 함 (대소문자 구분)
- 변형은 허용되지 않음

**스킬 폴더 명명:**
- ✅ 케밥-케이스 사용: `notion-project-setup`
- ❌ 공백 없음: `Notion Project Setup`
- ❌ 밑줄 없음: `notion_project_setup`
- ❌ 대문자 없음: `NotionProjectSetup`

**README.md 금지:**
- 스킬 폴더 내에 README.md를 포함하지 마세요
- 모든 문서는 SKILL.md 또는 references/에 들어갑니다

> **참고**: GitHub를 통해 배포할 때는 사람 사용자를 위한 저장소 수준 README가 필요합니다
#### YAML 프론트매터: 가장 중요한 부분

YAML 프론트매터는 Claude가 스킬을 로드할지 결정하는 방법입니다. 이 부분을 정확히 작성하세요.

**최소 필수 형식:**

```yaml
---
name: your-skill-name
description: 무엇을 하는지. 사용자가 [구체적인 트리거 문구]라고 물을 때 사용하세요!
---
```

이게 다입니다.
#### 필드 요구사항

**name** (필수):
- 케밥-케이스만 허용
- 공백이나 대문자 없음
- 폴더 이름과 일치해야 함

**description** (필수):
- 둘 다 포함: 스킬이 하는 일 + 사용 시기 (트리거 조건)
- 1024자 미만
- XML 태그 없음 (< 또는 >)
- 사용자가 말할 수 있는 특정 작업 포함
- 해당하는 경우 파일 유형 언급

**license** (선택):
- 스킬을 오픈 소스로 만들 경우 사용
- 예시: MIT, Apache-2.0

**compatibility** (선택):
- 1-500자
- 환경 요구사항 표시 (의도된 제품, 필요한 시스템 패키지, 네트워크 접근 필요 등)

**metadata** (선택):
- 모든 사용자 정의 키-값 쌍
- 권장: author, version, mcp-server
**예시:**

```yaml
metadata:
  author: ProjectHub
  version: 1.0.0
  mcp-server: projecthub
```
#### 보안 제한

**프론트매터에서 금지:**
- XML 꺾쇠 괄호 (`<>`)
- 이름에 "claude" 또는 "anthropic"이 포함된 스킬 (예약됨)

## 3장: 테스트 및 반복

스킬은 다양한 수준의 엄격함으로 테스트할 수 있습니다:

- **스킬 API를 통한 프로그래밍적 테스트**: 정의된 테스트 케이스에 대해 체계적으로 실행되는 평가 스위트를 빌드하세요.

품질 요구사항과 스킬의 가시성에 맞는 접근 방식을 선택하세요. 소규모 팀에서 내부적으로 사용되는 스킬은 수천 명의 엔터프라이즈 사용자에게 배포되는 스킬과 다른 테스트 요구사항이 있습니다.

### 프로 팁: 확장 전 단일 작업 반복

가장 효과적인 스킬 생성자는 Claude가 성공할 때까지 단일 도전적인 작업을 반복한 다음, 성공한 접근 방식을 스킬로 추출하는 것을 발견했습니다. 이는 Claude의 인컨텍스트 학습을 활용하고 광범위한 테스트보다 더 빠른 신호를 제공합니다. 작동하는 기반이 있으면 안정성을 위해 여러 테스트 케이스로 확장하세요.

### 권장 테스트 접근 방식
초기 경험을 바탕으로, 효과적인 스킬 테스트는 일반적으로 세 가지 영역을 다룹니다:

#### 1. 트리거링 테스트

**목표**: 스킬이 적절한 시간에 로드되는지 보장

**테스트 케이스:**
- ✅ 분명한 작업에서 트리거
- ✅ 재구성된 요청에서 트리거
- ❌ 관련 없는 주제에서 트리거되지 않음

**예시 테스트 스위트:**

```
트리거되어야 함:
- "새로운 ProjectHub 작업 공간 설정을 도와줘"
- "ProjectHub에서 프로젝트를 생성해야 해"
- "Q4 계획을 위한 ProjectHub 프로젝트 초기화"

트리거되면 안 됨:
- "샌프란시스코의 날씨는 어떤가요?"
- "파이썬 코드 작성을 도와줘"
- "스프레드시트 생성" (ProjectHub 스킬이 시트를 처리하지 않는 한)
```
#### 2. 기능 테스트

**목표**: 스킬이 올바른 출력을 생성하는지 확인

**테스트 케이스:**
- 유효한 출력 생성
- API 호출 성공
- 오류 처리 작동
- 엣지 케이스 커버

**예시:**

```
테스트: 5개 작업으로 프로젝트 생성

주어진: 프로젝트 이름 "Q4 Planning", 5개 작업 설명
실행: 스킬이 워크플로우 실행
결과:
- ProjectHub에 프로젝트 생성됨
- 올바른 속성으로 5개 작업 생성됨
- 모든 작업이 프로젝트에 연결됨
- API 오류 없음
```

#### 3. 성능 비교

**목표**: 스킬이 결과를 개선하는지 입증

**기준 비교:**

| 항목 | 스킬 없음 | 스킬 있음 |
|---|---|---|
| 사용자 입력 | 매번 지침 제공 | 자동 워크플로우 실행 |
| 상호작용 | 15번의 왕복 메시지 | 2개의 명확화 질문만 |
| API 호출 실패 | 3개 (재시도 필요) | 0개 |
| 토큰 소비 | 12,000개 | 6,000개 |

#### 반복적 개선

#### 사용 방법

```
skill-creator 스킬을 사용하여 [사용 사례]에 대한 스킬을 빌드하는 것을 도와줘
```

> **참고**: skill-creator는 스킬을 설계하고 정제하는 데 도움이 되지만 자동화된 테스트 스위트를 실행하거나 정량적 평가 지표를 생성하지는 않습니다.

### API를 통한 스킬 사용

프로그래밍적 사용 사례의 경우 (스킬을 활용하는 애플리케이션, 에이전트 또는 자동화된 워크플로우 구축 등), API는 스킬 관리 및 실행에 대한 직접적인 제어를 제공합니다.

#### 핵심 기능

- `/v1/skills` 엔드포인트로 스킬 나열 및 관리

- Claude 콘솔을 통한 버전 제어 및 관리
- 커스텀 에이전트 구축을 위한 Claude Agent SDK와 함께 작동
#### API를 통해 스킬을 사용할 때

| 사용 사례 | 최적의 표면 |
|---|---|
| 스킬과 직접 상호작용하는 최종 사용자 | Claude.ai / Claude Code |
| 개발 중 수동 테스트 및 반복 | Claude.ai / Claude Code |
| 개인적, 임시 워크플로우 | Claude.ai / Claude Code |
| 프로그래밍적으로 스킬을 사용하는 애플리케이션 | API |
| 대규모 프로덕션 배포 | API |
| 자동화된 파이프라인 및 에이전트 시스템 | API |

> **참고**: API의 스킬은 스킬에 필요한 보안 환경을 제공하는 코드 실행 도구 베타가 필요합니다.

```markdown
- MCP 서버가 연결되어 있는지 확인

### 4. 테스트

Claude에게 물어보세요: "[서비스]에서 새 프로젝트 설정"
```

### 스킬 포지셔닝

스킬을 설명하는 방식이 사용자가 가치를 이해하고 실제로 사용할지 결정합니다. README, 문서 또는 마케팅에서 스킬에 대해 작성할 때 다음 원칙을 염두에 두세요:

#### 결과에 집중, 기능이 아닌

**✅ 좋음:**

> ProjectHub 스킬은 팀이 페이지, 데이터베이스, 템플릿을 포함한 완전한 프로젝트 작업 공간을 30분의 수동 설정 대신 몇 초 만에 설정할 수 있도록 합니다.

**❌ 나쁨:**

> ProjectHub 스킬은 YAML 프론트매터와 Markdown 지침을 포함하여 MCP 서버를 호출하는 폴더입니다.

#### MCP + 스킬 이야기 강조

> 우리의 MCP 서버는 Claude에게 Linear 프로젝트에 대한 접근 권한을 제공합니다. 우리의 스킬은 Claude에게 팀의 스프린트 계획 워크플로우를 가르칩니다. 함께 AI 기반 프로젝트 관리를 가능하게 합니다.

## 5장: 패턴 및 문제 해결

이 패턴은 얼리 어답터와 내부 팀이 만든 스킬에서 나타난 것입니다. 규범적 규칙보다는 잘 작동하는 일반적인 접근 방식을 나타냅니다.

### 접근 방식 선택: 문제 중심 vs 도구 중심
홈디포(Home Depot) 예시를 생각해보세요:
- 고객이 "주방 캐비닛을 고쳐야 해요"라고 말하면, 직원이 올바른 도구를 알려줍니다.
- 또는 고객이 새 드릴을 꺼내 특정 작업에 어떻게 사용할지 물어볼 수 있습니다.

스킬도 같은 방식으로 작동합니다:

---

#### 문제 중심 접근

**사용자**: "프로젝트 작업 공간을 설정해야 해요"

**스킬**: 올바른 순서로 올바른 MCP 호출을 오케스트레이션합니다.

**역할 분담**:
- 사용자는 결과를 설명
- 스킬이 도구를 처리

#### 도구 중심 접근

**사용자**: "Notion MCP가 연결되어 있어요"

**스킬**: Claude에게 최적의 워크플로우와 모범 사례를 가르칩니다.

**역할 분담**:
- 사용자는 접근 권한 보유
- 스킬이 지침 제공

> 대부분의 스킬은 한 방향으로 기울어집니다. 사용 사례에 맞는 프레이밍이 무엇인지 아는 것은 올바른 패턴을 선택하는 데 도움이 됩니다.

---

### 패턴 1: 순차적 워크플로우 오케스트레이션

**사용 시기**: 특정 순서가 필요한 다단계 프로세스

**예시 구조:**

```markdown
## 워크플로우: 신규 고객 온보딩

### 1단계: 계정 생성
- MCP 도구 호출: `create_customer`
- 매개변수: name, email, company

### 2단계: 결제 설정
- MCP 도구 호출: `setup_payment_method`
- 대기: 결제 수단 확인

### 3단계: 구독 생성
- MCP 도구 호출: `create_subscription`
- 매개변수: plan_id, customer_id (1단계에서)

### 4단계: 환영 이메일 전송
- MCP 도구 호출: `send_email`
- 템플릿: welcome_email_template
```

**핵심 기법:**
- 명시적 단계 순서
- 단계 간 의존성
- 각 단계의 검증
- 실패 시 롤백 지침

---

### 패턴 2: 다중 MCP 조정

**사용 시기**: 워크플로우가 여러 서비스에 걸칠 때

**예시: 디자인-투-개발 핸드오프**

```markdown
### 1단계: 디자인 내보내기 (Figma MCP)
1. Figma에서 디자인 자산 내보내기
2. 디자인 사양 생성
3. 자산 매니페스트 생성

### 2단계: 자산 저장 (Drive MCP)
1. Drive에 프로젝트 폴더 생성
2. 모든 자산 업로드
3. 공유 가능한 링크 생성

### 3단계: 작업 생성 (Linear MCP)
1. 개발 작업 생성
2. 작업에 자산 링크 첨부
3. 엔지니어링 팀에 할당

### 4단계: 알림 (Slack MCP)
1. #엔지니어링에 핸드오프 요약 게시
2. 자산 링크 및 작업 참조 포함
```

**핵심 기법:**
- 명확한 단계 분리
- MCP 간 데이터 전달
- 다음 단계로 이동하기 전 검증
- 중앙 집중식 오류 처리

---

### 패턴 3: 반복적 정제

**사용 시기**: 피드백으로 출력 품질이 향상될 때

**예시: 보고서 생성**

```markdown
## 반복적 보고서 생성

### 초안 작성
1. MCP를 통해 데이터 가져오기
2. 보고서 초안 생성
3. 임시 파일에 저장

### 품질 검사
검증 스크립트 실행:
- 누락된 섹션 식별
- 일관되지 않은 형식
- 데이터 검증 오류

### 정제 루프
1. 식별된 각 문제 해결
2. 영향을 받는 섹션 재생성
3. 재검증
4. 품질 임계값에 도달할 때까지 반복

### 최종 확정
1. 최종 형식 적용
2. 요약 생성
3. 최종 버전 저장
```

**핵심 기법:**
- 명시적 품질 기준
- 반복적 개선
- 검증 스크립트
- 언제 반복을 멈플지 알기

### 패턴 4: 컨텍스트 인식 도구 선택

**사용 시기**: 동일한 결과이지만 컨텍스트에 따라 다른 도구

**예시: 파일 저장**

```markdown
## 스마트 파일 저장

### 결정 트리
1. 파일 유형 및 크기 확인
2. 최적의 저장 위치 결정:
   - 대용량 파일 (>10MB): 클라우드 저장 MCP 사용
   - 협업 문서: Notion/Docs MCP 사용
   - 코드 파일: GitHub MCP 사용
   - 임시 파일: 로컬 저장 사용

### 저장 실행
결정에 기반하여:
- 적절한 MCP 도구 호출
- 서비스별 메타데이터 적용
- 접근 링크 생성

### 사용자에게 컨텍스트 제공
해당 저장이 선택된 이유 설명
```

- 명확한 결정 기준
- 대체 옵션
- 선택에 대한 투명성

**사용 시기**: 스킬이 도구 접근을 넘어 전문화된 지식을 추가할 때

**예시: 금융 규정 준수**

## 문제 해결

### 스킬이 업로드되지 않음

**오류**: "업로드된 폴더에서 SKILL.md를 찾을 수 없습니다"

**원인**: 파일이 정확히 SKILL.md로 명명되지 않음

**해결책:**
1. SKILL.md로 이름 변경 (대소문자 구분)
2. 확인: `ls -la`는 SKILL.md를 표시해야 함

---

### 오류: "잘못된 프론트매터"

**원인**: YAML 형식 문제

**일반적인 실수:**

```yaml
# ❌ 잘못됨 - 구분자 누락
name: my-skill
description: Does things

# ❌ 잘못됨 - 닫히지 않은 인용부호
name: my-skill
description: "Does things

# ✅ 올바름
---
name: my-skill
description: Does things
---
```

---

### 오류: "잘못된 스킬 이름"

**원인**: 이름에 공백이나 대문자가 있음

```yaml
# ❌ 잘못됨
name: My Cool Skill

# ✅ 올바름
name: my-cool-skill
```

---

### 스킬이 트리거되지 않음

**증상**: 스킬이 자동으로 로드되지 않음

**해결책**: description 필드 수정

**빠른 체크리스트:**
- 너무 일반적인가요? ("프로젝트를 도와줍니다"는 작동하지 않음)
- 사용자가 실제로 말할 트리거 문구를 포함하나요?
- 해당하는 경우 관련 파일 유형을 언급하나요?

**디버깅 접근법:**

Claude에게 물어보세요: "[스킬 이름] 스킬을 언제 사용하나요?"

Claude가 설명을 인용하여 다시 말할 것입니다. 누락된 것을 기반으로 조정하세요.

---

### 스킬이 너무 자주 트리거됨

**증상**: 관련 없는 쿼리에 대해 스킬 로드

**해결책 1: 부정적 트리거 추가**

```yaml
description: CSV 파일을 위한 고급 데이터 분석. 통계 모델링, 회귀, 클러스터링에 사용.
  단순한 데이터 탐색에는 사용하지 마세요 (data-viz 스킬 사용).
```

**해결책 2: 더 구체적으로 작성**

```yaml
# ❌ 너무 광범위
description: 문서 처리

# ✅ 더 구체적
description: 계약 검토를 위한 PDF 법적 문서 처리
```

**해결책 3: 범위 명확화**

```yaml
description: 전자상거래를 위한 PayFlow 결제 처리. 온라인 결제 워크플로우에 특별히 사용,
  일반적인 금융 분석에는 사용하지 않음.
```

---

### MCP 연결 문제

**증상**: 스킬은 로드되지만 MCP 호출 실패

**체크리스트:**

1. **MCP 서버가 연결되어 있는지 확인**
   - Claude.ai: 설정 > 확장 프로그램 > [서비스]
   - "연결됨" 상태 표시되어야 함

2. **인증 확인**
   - API 키가 유효하고 만료되지 않았는지
   - 적절한 권한/범위가 부여되었는지
   - OAuth 토큰이 갱신되었는지

3. **독립적으로 MCP 테스트**
   - Claude에게 직접 MCP 호출을 요청하세요 (스킬 없이)
   - "[서비스] MCP를 사용하여 내 프로젝트 가져오기"
   - 이것이 실패하면 문제는 MCP이지 스킬이 아님

4. **도구 이름 확인**
   - 스킬이 올바른 MCP 도구 이름을 참조하는지 확인
   - MCP 서버 문서 확인
   - 도구 이름은 대소문자 구분

---

### 지침이 따르지 않음

**증상**: 스킬은 로드되지만 Claude가 지침을 따르지 않음

#### 일반적인 원인

**1. 지침이 너무 장황함**
- 지침을 간결하게 유지
- 글머리 기호와 번호 목록 사용
- 자세한 참조를 별도 파일로 이동

**2. 지침이 묻힘**
- 중요한 지침을 상단에 배치
- `## 중요` 또는 `## 필수` 헤더 사용
- 필요한 경우 핵심 포인트 반복

**3. 모호한 언어**

```markdown
# ❌ 나쁨
적절히 검증하세요

# ✅ 좋음
필수: create_project 호출 전에 확인하세요:
- 프로젝트 이름이 비어 있지 않음
- 최소 한 명의 팀원이 할당됨
- 시작 날짜가 과거가 아님
```
#### 고급 기법

중요한 검증의 경우, 언어 지침에 의존하기보다 프로그래밍적으로 검사를 수행하는 스크립트를 번들로 포함하는 것을 고려하세요. 코드는 결정론적입니다. 언어 해석은 그렇지 않습니다.

---

### 모델 "게으름"

**해결책**: 명시적인 격려 추가

```markdown
## 성능 참고사항
- 철저히 수행할 시간을 가지세요
- 속도보다 품질이 더 중요합니다
- 검증 단계를 건너뛰지 마세요
```

> **참고**: 사용자 프롬프트에 이것을 추가하는 것이 스킬에 추가하는 것보다 더 효과적입니다.

### 대용량 컨텍스트 문제

**증상**: 스킬이 느리거나 응답이 저하됨

**원인:**
- 스킬 콘텐츠가 너무 큼
- 너무 많은 스킬이 동시에 활성화됨
- 점진적 공개 대신 모든 콘텐츠가 로드됨

**해결책:**

**1. SKILL.md 크기 최적화**
- 자세한 문서를 references/로 이동
- 인라인 대신 참조에 링크
- SKILL.md를 5,000단어 미만으로 유지

**2. 활성화된 스킬 감소**
- 동시에 20-50개 이상의 스킬이 활성화되어 있는지 평가
- 선택적 활성화 권장
- 관련 기능을 위한 스킬 "팩" 고려
