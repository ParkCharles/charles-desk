# GitHub Pages Markdown 작성 가이드라인

이 문서는 GitHub Pages에서 읽기 쉬운 글을 작성하기 위한 가이드라인이다.
LLM이 글을 생성할 때도 참고할 수 있도록 규칙 중심으로 작성한다.

---

## 1. 기본 작성 원칙

- 가독성이 최우선이다
- 비전공자, 고령자도 읽을 수 있는 수준을 목표로 한다
- 한 문단은 3~4줄 이내로 유지한다
- 한 문장에는 하나의 메시지만 담는다
- 설명 가능한 내용은 문장보다 리스트를 우선 사용한다

---

## 2. 파일 및 메타 정보 규칙

### 파일명 규칙
- 형식: `YYYY-MM-DD-제목.md`
- 영문 소문자와 하이픈만 사용한다
- 공백과 한글은 사용하지 않는다
- 예: `2025-01-13-markdown-guide.md`

### Front Matter 템플릿
모든 문서 상단에 아래 형식을 포함한다:

```yaml
---
layout: post
title: "문서 제목"
date: 2025-01-13
categories: [카테고리명]
toc: true
---
```

---

## 3. 제목 구조 규칙

- 문서 전체 제목은 `#` 1개만 사용한다
- 큰 주제는 `##`로 시작한다
- 세부 설명은 `###`까지만 사용한다
- `####`는 가급적 사용하지 않는다

### 예시 구조

```markdown
# 문서 제목
## 왜 이 주제를 다루는가
### 문제 상황
### 해결 아이디어
## 결론
```

---

## 4. 문단 작성 규칙

- 문단과 문단 사이에는 반드시 한 줄을 비운다
- 하나의 문단에는 하나의 핵심만 포함한다
- 길어질 경우 리스트로 분해한다

### 나쁜 예
이 서비스는 사용하기 쉽고 직관적이며 확장성까지 고려하여 설계되었다.

### 좋은 예
- 사용하기 쉬운 구조
- 직관적인 화면 구성
- 향후 확장 가능성 고려

---

## 5. 리스트 사용 규칙

- 순서 없는 항목은 하이픈(`-`) 사용
- 순서가 중요한 경우 숫자 리스트 사용
- 중첩 리스트는 최대 2단계까지만 허용
- 리스트 항목이 7개를 초과하면 그룹으로 분리한다

---

## 6. 강조 표현 가이드

| 표현 방식 | 용도 | 예시 |
|-----------|------|------|
| `**굵게**` | 반드시 기억해야 할 핵심 개념 | **중요한 내용** |
| `*기울임*` | 맥락 설명, 부연 설명 | *참고로* |
| `` `코드` `` | 기술 용어, API명, 변수명 | `config.yml` |

- 한 문단에 강조 표현은 1~2개까지만 사용한다
- 강조가 많으면 아무것도 강조되지 않는다

---

## 7. 테마 및 스타일 설정

### 권장 테마
GitHub Pages 기본 제공 테마 중 가독성이 좋은 테마를 선택한다:
- `minima`: 심플하고 읽기 편함 (기본 권장)
- `cayman`: 헤더가 강조된 스타일
- `slate`: 다크 계열 선호 시

### _config.yml 설정 예시

```yaml
theme: minima
title: "블로그 제목"
description: "블로그 설명"
```

### 스타일 커스터마이징
- 기본 테마를 유지하되 필요시 `_sass/` 폴더에서 수정
- 본문 폰트 크기는 16px 이상 유지
- 줄 간격(line-height)은 1.6 이상 권장

---

## 8. 이미지 삽입 규칙

### 기본 형식

```markdown
![이미지 설명](이미지경로)
*이미지 캡션은 바로 아래에 기울임으로 작성*
```

### 규칙
- 모든 이미지에 alt 텍스트를 반드시 포함한다
- 이미지 파일명은 영문 소문자와 하이픈만 사용한다
- 큰 이미지는 가로 800px 이하로 리사이즈한다
- 스크린샷은 불필요한 영역을 잘라낸다

---

## 9. 표(Table) 사용 규칙

- 3열 이하의 단순 비교에만 사용한다
- 복잡한 데이터는 리스트로 분해한다
- 셀 내용은 짧게 유지한다

### 예시

| 항목 | 설명 |
|------|------|
| 이름 | 사용자 이름 |
| 이메일 | 연락용 이메일 |

---

## 10. 코드 설명 규칙

- 설명을 먼저 쓰고 그 다음에 코드를 배치한다
- 코드의 목적이 한 줄로 명확히 설명되어야 한다
- 코드 블록에는 언어를 명시한다

### 예시
아래 코드는 설정 파일에서 테마를 지정하는 방법이다:

```yaml
theme: minima
```

---

## 11. 접근성 고려사항

- 이미지에는 반드시 alt 텍스트를 포함한다
- 링크 텍스트는 구체적으로 작성한다
  - 나쁜 예: "여기를 클릭하세요"
  - 좋은 예: "마크다운 공식 문서 보기"
- 색상만으로 정보를 구분하지 않는다
- 중요한 정보는 굵은 글씨나 구조로 강조한다

---

## 12. 이모지 사용 규칙

- 섹션 시작 또는 주의 환기에만 사용한다
- 문장 중간에 이모지를 남발하지 않는다

### 허용되는 용도
- ✅ 완료, 권장 사항
- ⚠️ 주의 사항
- 💡 팁, 아이디어
- ❌ 금지, 나쁜 예시

---

## 13. 문서 마무리 규칙

- 모든 문서는 요약 섹션으로 마무리한다
- 요약은 3줄 이내로 작성한다
- 독자가 다음에 무엇을 하면 좋은지 안내한다

### 마무리 템플릿

```markdown
## 요약
- 핵심 내용 1
- 핵심 내용 2
- 핵심 내용 3

## 다음 단계
- 이 가이드라인을 기준으로 첫 문서를 작성해본다
- 작성 후 모바일에서 가독성을 확인한다
```

---

## 체크리스트

문서 작성 완료 후 아래 항목을 확인한다:

- [ ] 파일명이 `YYYY-MM-DD-제목.md` 형식인가
- [ ] Front Matter가 포함되어 있는가
- [ ] `#` 제목이 하나만 있는가
- [ ] 문단이 3~4줄을 넘지 않는가
- [ ] 모든 이미지에 alt 텍스트가 있는가
- [ ] 강조 표현이 과하지 않은가
- [ ] 요약 섹션이 있는가
